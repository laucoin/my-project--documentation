# User

A **User** is an application account — a person who can log in and interact with the platform. Users are distinct from
[Participants](/functional/business-objects/participant): a user is the identity that operates the system, while a
participant is a person tracked within a project.

::: info User ≠ Participant
A user and a participant are separate concepts. The same individual can be both (e.g. a youth participant who also has
an app account), but the two records remain independent. Linking them is optional and never mandatory.
:::

## Scope

Every user belongs to exactly one **Organisation**. A user's email must be unique within their organisation — the same
email address can exist in two different organisations without conflict.

## Identity and authentication

User accounts are backed by **Keycloak**. The `oidc_id` field stores the Keycloak user UUID, which links the
application record to the IdP identity. First name, last name, and email are synchronised from the Keycloak token on
first login.

## Global role

Every user holds exactly one global role:

| Role          | Description                                                           |
|---------------|-----------------------------------------------------------------------|
| `USER`        | Standard role — assigned by default to every account                  |
| `SUPER_ADMIN` | Platform administrator — full visibility across the entire platform   |

For a detailed description of what each role can do, see [Roles](/functional/roles).

## Project access via profiles

A user gains access to a project through a **profile**. A profile links the user to a specific project, assigns a
project-level role, and defines the period during which access is active. A user can hold at most one profile per
project.

See [Roles → Profiles](/functional/roles#profiles) for the full lifecycle.

## Key attributes

| Attribute         | Description                                                                                   |
|-------------------|-----------------------------------------------------------------------------------------------|
| `id`              | Primary key — UUID generated by the application                                               |
| `organization_id` | FK → `organizations.id` — owning organisation; part of composite unique key with `email`     |
| `oidc_id`         | Keycloak user UUID — links the record to the IdP identity                                     |
| `first_name`      | Populated from the Keycloak token on first login *(nullable — absent before first login)*     |
| `last_name`       | Populated from the Keycloak token on first login *(nullable)*                                 |
| `email`           | Unique within the organisation — used as the login identifier; composite unique key with `organization_id` |
| `role`            | Global role: `USER` or `SUPER_ADMIN`                                                          |
| `last_login`      | Timestamp of the most recent successful authentication                                        |
| `created_at`      | Creation timestamp — managed by Spring Auditing                                               |
| `created_by`      | FK → `users.id` — user who created the record                                                 |
| `updated_at`      | Last modification timestamp — managed by Spring Auditing                                      |
| `updated_by`      | FK → `users.id` — user who last modified the record                                           |
| `purged_at`       | GDPR purge timestamp — personal data erased; shell kept for audit integrity                   |
| `deleted_at`      | Soft-delete timestamp — `NULL` means active                                                   |

## Relationships

| Related object | Relationship                                              |
|----------------|-----------------------------------------------------------|
| Organisation   | A user belongs to one organisation                        |
| Profile        | A user can hold zero or more project profiles             |
| Participant    | A user can optionally be linked to one participant per project |

## Data lifecycle

### Soft delete

Deleting a user sets `deleted_at` — the account is deactivated but the record is retained for audit and data integrity.

### GDPR purge

A user can request the permanent deletion of their personal data. When purged, personal information (name, email,
`oidc_id`) is erased and `purged_at` is set. The account shell is kept to preserve the audit trail on records the user
created or modified.

::: warning Purge constraints
A user cannot be purged if they are the sole permanent `PROJECT_ADMIN` on one or more projects. They must first
transfer that role or delete the projects. See [Data Policy](/functional/data-policy) for the full rules.
:::
