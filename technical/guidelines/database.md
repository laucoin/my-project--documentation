# Database Conventions

The application uses a single **PostgreSQL** instance with three isolated schemas — one per domain module.
The query builder is **jOOQ** (R2DBC dialect). Flyway manages migrations (JDBC, startup only).

---

## Schema Ownership

**Convention: each module owns exactly one schema. No module may access another module's schema.**

| Module       | Schema         |
|--------------|----------------|
| Core         | `core`         |
| Operation    | `operation`    |
| Registration | `registration` |

Rules (from [ADR 003](/technical/adr/003-schema-per-module)):

- A module's repositories only access tables in their own schema
- Cross-schema queries are forbidden — even via raw SQL or jOOQ
- Cross-module data needs must go through the owning module's inbound service port
- Join queries across schema boundaries are not permitted at the database layer

```kotlin
// ✅ correct — each module queries only its schema
dslContext.select().from(CORE.PROJECTS).where(...)      // in core module
dslContext.select().from(OPERATION.MOVEMENTS).where(...) // in operation module

// ❌ forbidden — operation module querying core schema
dslContext.select().from(CORE.PROJECTS).join(OPERATION.MOVEMENTS)...
```

## Table Naming

**Convention: plural, snake_case.**

```sql
-- ✅ correct
CREATE TABLE projects (...);
CREATE TABLE group_members (...);
CREATE TABLE registration_periods (...);

-- ❌ avoid
CREATE TABLE project (...);
CREATE TABLE GroupMember (...);
```

**Rationale:** A table represents a set of rows. Plural is semantically accurate and aligns with standard SQL
conventions.
The key rule is **consistency across all modules**.

## Identifiers

**Convention: UUID generated by the application, not the database.**

```sql
-- ✅ correct — DB only stores, app generates
id UUID NOT NULL,

-- ❌ avoid — DB-generated UUID creates issues with Spring Data R2DBC
id UUID NOT NULL DEFAULT uuid_generate_v4(),
```

**Rationale:** Spring Data R2DBC determines whether to issue an `INSERT` or `UPDATE` by checking if the `@Id` field is
`null`.
If the database generates the UUID, the entity arrives with `id = null` and Spring will attempt an `UPDATE` instead of
an `INSERT`.

In Kotlin:

```kotlin
// Service layer
val entity = ProjectEntity(id = UUID.randomUUID(), ...)
repository.save(entity)
```

## Timestamps

**Convention: `created_at` and `updated_at`, managed by the application via Spring Data Auditing.**

```sql
-- ✅ correct — no DEFAULT, no trigger
created_at TIMESTAMPTZ NOT NULL,
updated_at TIMESTAMPTZ NOT NULL,
```

```kotlin
// ✅ correct — managed by @EnableR2dbcAuditing
@CreatedDate
val createdAt: OffsetDateTime,

@LastModifiedDate
val updatedAt: OffsetDateTime,
```

**Rationale:** Database triggers and `DEFAULT NOW()` are redundant when Spring Auditing is enabled.
Keeping logic in the application layer makes the codebase more portable and testable.

> Enable auditing in each module's database config class with `@EnableR2dbcAuditing`.

## Column Naming

**Convention: snake_case, explicit and descriptive.**

```sql
-- ✅ correct
first_name          TEXT        NOT NULL,
is_active           BOOLEAN     NOT NULL DEFAULT TRUE,
organisation_id     UUID        NOT NULL,

-- ❌ avoid
firstName           TEXT,       -- camelCase in SQL
flag                BOOLEAN,    -- too vague
org_id              UUID,       -- abbreviation
```

## Constraints Naming

**Convention: explicit constraint names with a prefix.**

| Type        | Prefix | Example                          |
|-------------|--------|----------------------------------|
| Primary Key | `pk_`  | `pk_projects`                    |
| Unique      | `uq_`  | `uq_projects_slug`               |
| Foreign Key | `fk_`  | `fk_group_members_project_id`    |
| Index       | `idx_` | `idx_projects_organisation_id`   |
| Check       | `ck_`  | `ck_registration_periods_status` |

```sql
CONSTRAINT pk_projects                 PRIMARY KEY (id),
CONSTRAINT uq_projects_slug            UNIQUE (slug),
CONSTRAINT fk_group_members_project    FOREIGN KEY (project_id) REFERENCES projects(id)
```

## jOOQ Conventions

**Convention: always use jOOQ generated types for runtime queries. Raw SQL strings are forbidden.**

jOOQ classes are generated from the database schema after each migration. They provide compile-time safety and automatic
schema name embedding.

```kotlin
// ✅ correct — type-safe, compile-time checked
fun findByOrganisation(organisationId: UUID): Flow<ProjectRecord> =
    dslContext
        .selectFrom(PROJECTS)
        .where(PROJECTS.ORGANISATION_ID.eq(organisationId))
        .asFlow()

// ❌ avoid — raw SQL, no safety, schema name hardcoded
dslContext.fetch("SELECT * FROM core.projects WHERE organisation_id = '$organisationId'")
```

**Rules:**

- Run jOOQ code generation after every migration before committing
- Never hardcode schema names as strings — use the generated constants
- Keep jOOQ DSL calls in the `infrastructure/` layer only — never in domain or inbound port code
- For coroutine integration use: `asFlow()`, `awaitSingle()`, `awaitFirstOrNull()`, `awaitSingleOrNull()`

## Flyway Migrations

**Convention: date-based versioning with sequence suffix.**

```
V20260401_001__init_projects.sql
V20260401_002__add_project_slug.sql
V20260512_001__add_project_status.sql
```

Format: `V{YYYYMMDD}_{sequence}__{description}.sql`

**Rules:**

- Each module owns its migrations under `src/main/resources/db/migration/`
- Each module has its own Flyway history table: `flyway_{module}_history`
- Each module targets its own schema (set via `defaultSchema` in Flyway config)
- Never modify an already-applied migration — always create a new one
- Migration files are immutable once committed
- Run jOOQ code generation after applying a migration

**Rationale:** Date-based versions avoid numbering conflicts between modules and make the history readable at a glance.
