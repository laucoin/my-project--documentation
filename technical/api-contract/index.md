## General conventions

All endpoints follow the Reference Card:

- URLs use **plural nouns** in English тАФ `/participants`, not `/participant` or `/getParticipants`.
- Query parameters use **lower camelCase** тАФ `projectId`, `pageSize`.
- The API is versioned at `/api/v2/` тАФ the full base path is `/api/v2/`.
- Sub-resources reflect ownership тАФ `/api/v2/projects/{projectId}/participants`.
- All dates and datetimes use **ISO 8601** тАФ `2026-04-08` or `2026-04-08T09:42:51Z`.
- All IDs are **UUIDs** generated by the application.
- All request and response bodies use `application/json`.

## Required headers

Every authenticated request **must** include the following headers in addition to the
`Authorization` (OAuth2 Bearer) header:

| Header                | Required    | Description                                                                                                                                                      |
|-----------------------|-------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `Authorization`       | Yes         | `Bearer <access_token>` тАФ OAuth2 token obtained via the BFF session                                                                                              |
| `X-Organization-Slug` | Yes         | The slug of the organisation the request is scoped to (e.g. `my-org`). Used by the BFF to resolve the Keycloak organisation and enforce project-level isolation. |
| `Content-Type`        | Yes (write) | `application/json` тАФ required on `POST`, `PUT`, `PATCH` requests with a body                                                                                     |
| `Accept`              | No          | Defaults to `application/json`                                                                                                                                   |

::: warning X-Organization-Slug is mandatory
A request without `X-Organization-Slug` is rejected with
`400 Bad Request`. This header is the primary mechanism for organisation isolation тАФ it is validated against the authenticated user's organisation before any resource access.
:::

### Example

```http
GET /api/v2/projects HTTP/1.1
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
X-Organization-Slug: acme-corp
Accept: application/json
```

## HTTP status codes

| Code  | Meaning                                                  |
|-------|----------------------------------------------------------|
| `200` | Success тАФ general read or update                         |
| `201` | Created тАФ resource successfully created (`POST`)         |
| `202` | Accepted тАФ async operation started                       |
| `204` | No content тАФ successful `DELETE`                         |
| `206` | Partial content тАФ paginated response                     |
| `400` | Bad request тАФ validation error or malformed input        |
| `401` | Unauthorized тАФ missing or invalid authentication         |
| `403` | Forbidden тАФ authenticated but insufficient permissions   |
| `404` | Not found тАФ resource does not exist                      |
| `405` | Method not allowed тАФ verb not supported on this resource |
